# 输出格式要求

请严格按以下顺序输出：

## 1. 任务理解
- 用简洁条目总结你对任务的理解

## 2. 设计/改造方案
- 说明整体思路
- 说明为什么这样拆分
- 说明如何保证可维护性

## 3. 文件结构
- 给出建议或最终文件树
- 标出新增、修改、保留的文件

## 4. 文件职责说明
- 逐个说明每个文件负责什么
- 说明模块之间如何协作

## 5. 最终代码
- 输出完整可用代码或精确修改后的文件内容
- 不要只给伪代码
- 不要只给概念性描述

## 6. 风险与验证点
- 列出可能风险
- 列出需要验证的行为、边界情况和测试点

## 补充规则
- 必须先阅读当前实现，再动手
- 基于现有代码继续工作，不要另起一套
- 如果任务较小，不要过度机械拆分
- 如果发现已有文件明显过大或职责混乱，应主动拆分
- 最终必须执行并汇报验证命令结果

## 默认验证要求
至少执行：
- `npm run typecheck`
- `npm run build`

如果涉及主进程、IPC、discovery、adapter、orchestrator 等逻辑，还要补充执行：
- `npm run build:main && node --test dist/main/orchestratorService.test.js`

如果验证失败：
- 先修复
- 再输出最终结果
