# 工程规则

## 总原则
以“长期可维护、可扩展、易协作”为第一优先级完成任务，而不是只追求快速产出。

## 必须遵守
1. 不要把所有代码堆进单个文件。
2. 不要在一个文件中混合页面/入口、业务逻辑、数据处理、工具函数、类型定义、配置常量。
3. 每个模块尽量单一职责。
4. 公共能力必须抽离，禁止复制粘贴逻辑。
5. 保持输入输出边界清晰，避免隐藏依赖。
6. 不要为了缩短代码而牺牲可读性。
7. 命名必须语义化。
8. 优先保持现有公开接口稳定。
9. 不要随意引入新依赖。
10. 输出结果必须适合团队继续维护。

## 可维护性要求
- 模块拆分清晰
- 文件职责单一
- 目录结构合理
- 类型边界明确
- 工具函数独立
- 配置与常量独立
- 复杂逻辑抽离
- 避免循环依赖
- 易于测试
- 修改范围可控、可回滚

## 文件组织偏好
- 页面/入口层：负责协调与组装
- hook / controller：负责状态与流程编排
- service：负责核心流程、文件读写、接口交互
- component：负责 UI 展示
- utils：纯函数
- shared / types：类型定义
- constants / config：常量与配置
- tests：关键测试

## 规模控制
- 单文件避免过大
- 单模块不要承担多个无关职责
- 发现文件明显过长或复杂时，主动拆分
- 如果某处不拆分，必须有合理理由

## 变更原则
1. 优先最小侵入修改
2. 保留现有业务语义
3. 保持对外接口兼容
4. 明确区分：功能新增 / 重构 / 问题修复
5. 标注风险点

## 禁止事项
1. 禁止一切都放进一个文件
2. 禁止只给概念，不给可执行结果
3. 禁止无边界重构整个工程
4. 禁止随意改变公共接口
5. 禁止重复逻辑散落多个文件
6. 禁止不必要抽象
7. 禁止巨型类、巨型函数、巨型组件
8. 禁止忽略异常处理、边界条件和基本验证
9. 禁止与任务无关的花哨优化
10. 禁止把“可维护性”等同于只加注释

## 技术约束
- TypeScript 严格模式
- TS 源码 import 保持 `.js` 后缀
- Electron-safe 路由
- main / preload / shared / renderer 边界不能破坏
- 不允许使用：
  - `as any`
  - `@ts-ignore`
  - `@ts-expect-error`
